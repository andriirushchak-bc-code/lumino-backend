Lumino — Technical Architecture

Reference documentation for the Lumino demo bot: a fully BEP-native AI Agent that showcases the platform's capability surface in a single 5-minute sales demo. Intended audience is technical reviewers, BotsCrew engineering, and anyone asked to extend or debug the bot.

Executive summary

What: Lumino is an AI Agent bot on the BotsCrew Enterprise Platform (BEP) that acts as an end-user assistant for a fictional B2B expense management SaaS. It embeds in the Lumino Expense web app as a chat widget.

Why: Every AI Action is deliberately mapped to a different BEP capability, so a single 5-minute demo exercises API calls, active context, generative composition, RAG, branching workflows, bulk writes, and multi-turn agentic reasoning without needing multiple bots.

Design principle: BEP-native only. No external LLMs. No external backends beyond the fictional app's database. ODIN does all reasoning, including the schema-inference work in Smart Import that an earlier draft delegated to Gemini.

Live demo Demo environment: lumino-backend…/demo. Widget embedded bottom-right. Demo user: andrii.bondar@northwind.example (hardcoded, no auth flow).

System architecture

Three tiers. The user talks to the widget, the widget talks to BEP, BEP talks to Supabase. There is no intermediate backend — the frontend shell is purely a visual host for the demo, not a gateway.

Fig. 1 — Three-tier architecture. Dashboard reads bypass BEP (they're client-side reads with the publishable key); all bot-driven reads and writes route through BEP workflows.

Why not a unified backend?

An earlier design placed a Next.js backend between BEP and Supabase (for Smart Import schema inference via Gemini). That was removed. Two reasons:

  • Architectural coherence: a BEP showcase bot should exercise BEP's capabilities, not hide them behind a second service.
  • ODIN is sufficient: once the Python step delivers a CSV sample as a string, ODIN reasons about columns and proposes a mapping inline — no second LLM call needed.

Request lifecycle

How a single user message becomes a reply. Reading this end-to-end clarifies where each BEP component acts.

Fig. 2 — Single-turn sequence. Steps 4–6 vary by step type: API_CALL hits Supabase, GET_KNOWLEDGE hits the KB, PYTHON_EXECUTOR computes locally.

The Knowledge reply atom

Every AI-generated reply surfaces through BEP's hardcoded Knowledge reply atom. It has no static message content — ODIN composes the response at runtime using last_ai_action_result (the terminal-step payload) plus the calling Action's RESPONSE FORMAT section. This is the single point where workflow output becomes user-visible text.

Contract Every workflow that produces a user-visible reply must terminate in either (a) a Python step that writes last_ai_action_result, or (b) a GET_KNOWLEDGE step with searchResultAttribute.name = "last_ai_action_result". Without this, the reply falls through to the Default atom.

ODIN routing

ODIN is BEP's routing LLM. It receives every user message along with a system prompt (the "ODIN prompt") that defines the bot's persona, rules, active-context behavior, and — critically — descriptions of all available AI Actions. For each turn, ODIN decides: answer from conversation alone, call an AI Action, or fall through to the Knowledge Base.

Priority order

message → ODIN

  ├─ matches an AI Action (by description + params fit)
  │     └─ extract params · populate attributes · trigger workflow
  │
  ├─ doesn't match but relates to indexed knowledge
  │     └─ fall through to Knowledge Base (rare in Lumino — policy_lookup action handles KB explicitly)
  │
  └─ doesn't match anything
        └─ Default reply atom (sets conversation_outcome = failed)

The ODIN prompt (v1.2)

~6,200 characters. Well under the 10,000 char BEP limit. Notable sections:

  • Persona & tone — first-person, professional, Lumino-specific
  • Active context rules — "the Uber one" resolves to the most recently discussed expense without re-asking
  • Silent-state rule (v1.2) — never narrate internal state changes like "ACTIVE EXPENSE is now set..." (learned from v1.0 testing)
  • Response style block (v1.2) — seven global rules: em-dash over double-hyphen, always lead tables with a sentence, single-bubble replies, sentence case, professional closings, specificity in composed content
  • URL suppression (v1.2) — global rule forbidding URLs in replies (all hardcoded placeholder links removed after testing surfaced broken lumino.example.com references)
  • Strict scope enforcement — off-topic queries redirected to expense domain
  • Escalation policy — explicit triggers for human handoff

The 9 AI Actions

Each AI Action maps 1:1 to a workflow. Actions are intent specs — their description is how ODIN decides to call them, and their params are how ODIN extracts inputs.

# Action Kind What it does Workflow summary
1 lookup_my_expenses READ Return user's N most recent expenses as a table GET /expenses with order=submitted_at.desc&limit={{requested_count}}
2 check_expense_status READ Status + approver for a specific expense — uses active context when available Conditional: if active_expense_id set, lookup by ID; else fuzzy match on expense_identifier
3 submit_expense WRITE Extract merchant/amount/category/date, confirm, write Python normalization (Kyiv-TZ date resolver) → POST /expenses
4 find_receipt READ Surface receipt URL or say none on file Keyword-tightened: single distinctive word extraction (e.g. "Berlin" from "the Berlin hotel")
5 policy_lookup RAG Answer policy questions citing section numbers GET_KNOWLEDGE against indexed policy chunks
6 explain_rejection READ + compose Show rejection reason; on follow-up, draft a professional appeal note GET rejected expense → compose appeal with merchant, amount, date, reason, user's defense
7 escalate_to_finance WRITE Open a support ticket; return reference ID (UUID) POST /tickets with Prefer: return=representation, extract id into last_ai_action_result
8 analyze_import_sheet AGENTIC Parse Google Sheet, have ODIN infer column mapping, flag ambiguities Python fetches gviz CSV → packages schema + sample rows → ODIN proposes mapping inline
9 confirm_import BULK WRITE Apply mapping (with user corrections) and bulk-insert rows Python re-fetches sheet → applies agreed mapping → 3 POSTs: /expenses, /guests, /import_sessions

Attribute wiring

41 bot attributes store intermediate state. Notable ones:

  • lumino_user_email — hardcoded to andrii.bondar@northwind.example, simulating a logged-in user
  • active_expense_id — set by lookup_my_expenses and check_expense_status, consumed by follow-up actions for silent context
  • requested_count — v1.1 fix: lets users ask for "last 3" vs "last 10"
  • knowledge_search_query — populated by policy_lookup params, fed into GET_KNOWLEDGE
  • last_ai_action_result — system attribute, the final payload every workflow must populate for Knowledge reply composition

Smart Import deep-dive

The signature capability. Two-action agentic flow that reads a Google Sheet, negotiates a column mapping in conversation, and bulk-imports on confirmation.

Fig. 3 — Smart Import two-turn flow. Between turns, the proposed mapping lives only in conversation memory — no PATCH endpoint or persistent draft state needed. ODIN updates the proposal in response to user corrections as plain language.

Why this design

Smart Import is the clearest illustration of BEP's agentic capability. ODIN isn't just picking tools in sequence — between turns 1 and 2, it holds a proposed mapping as conversational state, accepts natural-language corrections, and resolves them without re-running the analysis. The user says "map F&B to Entertainment" and ODIN updates its mental model; it does not need a mapping-edit endpoint, because the mapping isn't persisted yet.

BEP capabilities map

Every demo turn exercises a distinct BEP capability. This is the deliberate pedagogical structure of the bot.

Turn 1 · API call + table rendering

lookup_my_expenses: API_CALL GET, response extracted via JSONPath, rendered as markdown table in reply.

Turn 2 · Active context memory

check_expense_status: ODIN resolves "the Uber one" to active_expense_id set silently in turn 1.

Turn 3 · Conditional workflow

explain_rejection: workflow branches on whether active_expense_id is set or fuzzy lookup is needed.

Turn 4 · Generative composition

Same action, appeal-drafting branch: ODIN composes a professional note grounded in real expense data — no template.

Turn 5 · RAG over KB

policy_lookup: GET_KNOWLEDGE step against indexed policy chunks, returns §3.2 with 8-hour threshold reasoning.

Turn 6 · Agentic multi-turn reasoning

analyze_import_sheet: Python fetches, ODIN reasons about schema, flags F&B ambiguity — all in one turn, no external LLM.

Turn 7 · Branching workflow + bulk write

confirm_import: Python transforms, 3 parallel API_CALL POSTs to /expenses, /guests, /import_sessions.

Implicit · Human handoff

escalate_to_finance: creates a ticket, returns UUID reference — the demo-able "I'll get a human involved" moment.

Secrets & config

Secret key handling

BEP has no dedicated secret vault. The Supabase secret key is baked directly into the API_CALL headers block of each write step at import time. There are 5 POST steps that need the secret key:

  • submit_expense_flow — POST /expenses
  • escalate_to_finance_flow — POST /tickets
  • confirm_import_flow × 3 — POST /expenses, /guests, /import_sessions
Operational note The packaged zip contains the secret key in plaintext JSON. Do not commit it to public repositories or share outside the engineering team. If rotated, all 5 header entries must be updated via BEP's UI (or re-imported with a patched JSON).

Known gotchas

Google Sheet CSV URL

The Smart Import Python step uses /gviz/tq?tqx=out:csv rather than /export?format=csv. The export endpoint returns HTTP 400 for publicly-shared sheets without authentication; gviz works for "Anyone with link — Viewer." This is an undocumented Google quirk.

Date resolution timezone

The submit_expense_flow Python step uses a hardcoded UTC+3 offset (Europe/Kyiv summer time). This is not daylight-saving aware — during winter months it will be off by one hour, which still resolves to the correct calendar date 23 hours out of 24 but is technically fragile. A production deployment should use zoneinfo("Europe/Kyiv") or accept an explicit timezone attribute.

Supabase RLS

Row-Level Security is disabled on all tables for demo simplicity. Enabling RLS without adding policies will silently break read workflows (queries return empty arrays). If extending for production, add role-scoped policies and update the API_CALL headers to use user-scoped JWTs.

Widget iframe focus

The BEP widget embeds as a cross-origin iframe. Automated browser testing (e.g. Claude Cowork with Chrome extension) cannot reliably interact with the three-dot menu for session resets because of cross-origin focus restrictions. Manual testing is the current workaround; a keyboard shortcut or URL parameter for session reset would make automated QA significantly easier.

Prompt budget

ODIN prompt is at ~6,200 / 10,000 chars. Headroom for 2–3 more capability sections before needing to refactor. The AI Action descriptions count separately and are not part of this budget.
